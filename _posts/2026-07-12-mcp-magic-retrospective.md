> Figma와 AI를 연결하는 기술은 이미 있었다. 우리가 풀고 싶었던 문제는 사람들이 첫 성공까지 포기하지 않게 만드는 일이었다.

1년 동안 수만 명이 그 입구를 통과했다. 그러나 사용자가 늘수록 연결 이후의 운영이 더 큰 문제가 됐고, 우리가 풀던 경험은 공식 플랫폼 안으로 들어오기 시작했다.

Figma에서 AI가 디자인을 읽고 고치는 장면은 매력적이었다. 하지만 다른 사람에게 도구를 건네면 많은 이가 그 장면에 도달하기 전에 멈췄다. 런타임과 서버, MCP 클라이언트와 Figma 플러그인을 따로 설정하고, 문제가 생기면 터미널과 플러그인을 오가야 했기 때문이다. 그래서 흩어진 준비 과정을 하나의 데스크톱 앱에 담았다. 한국에서 유행하는 표현을 빌리면 **"딸깍"하면 서버가 뜨는 경험**을 만들고 싶었다.

2025년 7월, 공동 기여자인 **Yiseo, Jin**과 함께 첫 데스크톱 버전을 공개했다. 이후 MCP Magic이라는 이름으로 확장하고 앱 구조를 다시 썼다. 구현 방식은 여러 번 바뀌었지만 질문은 같았다.

> 어떻게 해야 더 많은 사람이 첫 연결을 지나 실제 작업까지 도달할 수 있을까?

## 첫 연결을 제품 경험으로 만들다

MCP Magic은 빈 곳에서 시작한 프로젝트가 아니다. 출발점에는 Grab Design 팀에서 사용하다 오픈소스로 공개한 [Cursor Talk To Figma MCP](https://github.com/grab/cursor-talk-to-figma-mcp)가 있었다. 자연어로 Figma를 읽고 수정하는 핵심 기능은 이미 작동하고 있었다.

우리가 풀어야 할 제품 문제는 핵심 기능보다 첫 성공까지의 경로였다. MCP Magic은 여러 도구에 흩어져 있던 서비스 실행, 설정 복사, 연결 상태와 로그를 한 화면으로 가져왔다.

앱이 MCP 자체를 더 강력하게 만든 것은 아니다. 이미 있던 힘에 도달하는 길을 짧고 분명하게 만들었다. 작은 차이처럼 보이지만, 첫 연결 전에 포기하면 강력한 기능도 존재하지 않는 것과 같았다.

## 실제 디자인 작업까지 이어졌다

그 길의 도달 범위는 생각보다 넓었다. 2025년 7월부터 1년 동안 GA4에는 **활성 사용자 31,360명**이 기록됐고, 앱에서는 MCP 도구가 총 **570,373회** 호출됐다. 한국에서 시작한 작은 데스크톱 앱이었지만 여러 언어권에서 사용됐다.

호출된 도구의 구성은 사용자가 먼저 Figma 문서와 선택한 요소를 읽고, 이어 텍스트와 프레임을 만들며 색과 형태를 다듬는 흐름을 시사했다. 성공 호출의 대부분이 조회보다 생성과 편집에 몰렸고, 최근 5개월 데이터에는 같은 세션에서 작업을 반복한 흔적도 남았다.

![디자인 목적별 MCP 성공 호출](/images/projects/2026-mcp-magic-retrospective/mcp-magic-mcp-calls-by-design-purpose.svg)

**사용자가 원한 것은 단순한 연결이 아니라 AI가 Figma를 실제로 만지는 순간이었다.** MCP Magic의 역할은 그 순간까지 도달할 수 있는 입구를 만드는 일이었다.

## 딸깍 다음에 남은 문제

사용자가 늘자 질문이 달라졌다.

> 서로 다른 사용자와 운영체제, MCP 클라이언트, 네트워크 환경에서도 계속 믿고 쓸 수 있게 만들 수 있는가?

Yiseo가 운영하는 채널에는 정말 "딸깍"만 하면 디자인이 되는지, AI는 모두 무료인지, Cursor나 Claude Code 설정까지 앱이 대신해 주는지 묻는 질문이 들어왔다.

사용자가 말하는 "안 된다"의 원인은 모두 달랐다. 서버나 포트, 플러그인의 채널, MCP 클라이언트 설정, 운영체제 보안 중 어디에서든 연결이 막힐 수 있었다.

원인은 달라도 사용자에게는 같은 말로 도착했다.

> 안 된다.

사용자의 숙련도도 예상보다 훨씬 넓었다. 어떤 사람에게는 IDE라는 말부터 낯설었고, 어떤 사람은 터미널을 여러 개 띄워 놓고 MCP를 깊게 사용했다. 개발자 도구와 디자인 도구의 경계가 빠르게 흐려지면서 서로 전혀 다른 기대를 가진 사람들이 한 제품 안으로 들어왔다.

첫 연결을 쉽게 만드는 것만으로는 충분하지 않았다. 그다음에는 사용자의 수준에 맞는 진단과 복구, 연결 안정성, 지원 운영이 필요했다. 작은 팀이 흥미만으로 감당하기에는 훨씬 넓은 문제였다.

## 플랫폼이 따라온 뒤

우리는 언어 지원을 넓히고, 별도의 IDE 없이 로컬 LLM으로 작업하는 방식도 시험했다. 그러나 시간이 갈수록 큰 업데이트를 계속할 이유는 약해졌다. 사용량이 없어서가 아니라, 우리가 풀던 핵심 경험이 공식 플랫폼 안으로 들어오기 시작했기 때문이다.

Figma는 자체 MCP 서버와 Agent를 통해 캔버스 안에서 디자인을 읽고 만들고 수정하는 흐름을 제공하기 시작했다. 사용자에게는 공식 기능이 가장 자연스럽고 친화적인 선택일 수 있다. 서드파티 앱이 같은 속도와 범위로 따라가는 데에는 분명한 한계가 있었다.

그 방향이 맞다는 판단과 만든 사람의 감정은 별개였다. 먼저 발견해 제품으로 만든 문제가 얼마 지나지 않아 더 큰 플랫폼의 기본 기능이 되는 경험은 반갑기만 하지 않았다. 무엇을 만들 수 있느냐보다, 무엇을 계속 붙잡아야 하느냐가 더 어려웠다.

그래서 MCP Magic은 종료보다 유지보수 모드에 가깝게 두려고 한다. 필요한 수정은 이어가겠지만, 지금의 형태로 큰 기능 업데이트를 계속할 이유는 이전보다 작아졌다.

## 다음에는 더 일찍 정할 것

돌이켜보면 첫 연결을 쉽게 만든 뒤의 결정을 오래 미뤘다.

첫째, **다음 목표를 더 빨리 좁혔어야 했다.** 설치 도우미로 남을지, 연결 진단과 복구 도구로 갈지, 반복 사용이 많았던 Figma 작업을 하나의 워크플로로 묶을지 정하지 못했다. 첫 번째 문제가 풀린 뒤에는 새로운 제품 가설이 필요했다.

둘째, **지원의 경계를 분명히 했어야 했다.** 어디까지가 앱의 책임이고 어디부터가 Figma 플러그인, MCP 클라이언트, 운영체제, AI 비용의 문제인지 더 명확해야 했다. 반복되는 "안 된다"를 자가 진단과 복구 흐름으로 바꾸는 일도 기능 개발만큼 중요했다.

셋째, **사용 데이터를 다음 가설에 더 일찍 연결했어야 했다.** 뒤늦게 데이터를 정리하고 나서야 사용자가 어디에서 연결을 시도하고, 실제 작업에 들어온 사용자가 어떤 생성과 편집을 반복하는지 보였다. 출시 초기부터 첫 연결, 첫 성공, 반복 사용을 나눠 봤다면 업데이트의 우선순위도 달라졌을 것이다.

이 세 가지는 부가적인 운영 업무가 아니었다. 첫 연결 다음에 만들어야 했던 제품 그 자체였다.

## 딸깍이 해결한 것, 남긴 것

이 프로젝트에서 가장 크게 배운 것은 사용자가 원하는 것이 강력한 기능만은 아니라는 점이다. 기능을 믿고 첫걸음을 뗄 수 있는 제품 경험도 필요했다. MCP Magic은 그 첫걸음을 "딸깍"으로 줄였고, 사용 데이터는 그 시도가 실제 사용자에게 닿았음을 보여줬다.

"딸깍"은 사용자를 0에서 80까지 데려갔지만, 마지막 20%에는 진단과 복구, 운영과 지원, 그리고 어디까지 책임질지에 대한 선택이 남았다. 다음에는 첫 성공 이후까지 무엇을 제품의 책임으로 둘지 더 일찍 정하려 한다.
