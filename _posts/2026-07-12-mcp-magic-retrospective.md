> 우리가 만든 것은 새로운 AI 기능이 아니라, 설치에서 첫 성공 호출까지 포기하지 않고 건널 수 있는 입구였다.

GA4에는 **31,360명의 활성 사용자**와 **570,373건의 MCP 도구 호출**이 남았다. 이 프로젝트는 작동했다. 그러나 사용자가 늘수록 연결 이후의 운영이 더 큰 문제가 됐고, 우리가 풀던 경험의 중심은 공식 플랫폼 안으로 이동하기 시작했다.

Figma에서 AI가 디자인을 읽고 고치는 장면은 매력적이었다. 하지만 다른 사람에게 도구를 건네면 많은 이가 그 장면에 도달하기 전에 멈췄다. 런타임과 서버, MCP 클라이언트와 Figma 플러그인을 따로 설정하고, 문제가 생기면 터미널과 플러그인을 오가야 했기 때문이다.

그래서 흩어진 준비 과정을 하나의 데스크톱 앱에 담았다. 한국에서 유행하는 표현을 빌리면 **"딸깍"하면 서버가 뜨는 경험**을 만들고 싶었다.

## 우리가 만든 것은 기능이 아니라 입구였다

MCP Magic은 빈 곳에서 시작한 프로젝트가 아니다. 출발점에는 Grab Design 팀에서 사용하다 오픈소스로 공개한 [Cursor Talk To Figma MCP](https://github.com/grab/cursor-talk-to-figma-mcp)가 있었다. 자연어로 Figma를 읽고 수정하는 핵심 기능은 이미 작동하고 있었다.

우리가 풀어야 할 제품 문제는 핵심 기능보다 첫 성공까지의 경로였다. MCP Magic은 여러 도구에 흩어져 있던 서비스 실행, 설정 복사, 연결 상태와 로그를 한 화면으로 가져왔다.

2025년 7월 18일 KMP 기반 첫 데스크톱 버전을 공개했고, 같은 해 10월 MCP Magic이라는 패밀리 앱으로 확장했다. 사용자 의견을 받은 뒤에는 Electron으로 구조를 다시 썼다. 이 과정은 공동 기여자인 **Yiseo, Jin**과 함께했다. 구현 방식은 여러 번 바뀌었지만 질문은 같았다.

> 어떻게 해야 더 많은 사람이 첫 연결을 지나 실제 작업까지 도달할 수 있을까?

앱이 MCP 자체를 더 강력하게 만든 것은 아니다. 이미 있던 힘에 도달하는 길을 짧고 분명하게 만들었다. 첫 연결 전에 포기하면 강력한 기능도 사용자에게는 존재하지 않는 것과 같았다.

## 이 프로젝트는 실제 작업으로 이어졌다

GA4의 실제 기록이 있는 11개월 동안 `mcp_tool_call` 이벤트를 한 번 이상 남긴 활성 사용자는 1,633명이었다. 앱 전체 활성 사용자 31,360명과는 이벤트별로 독립 집계된 값이므로, 31,360명이 모두 MCP 작업까지 갔다고 말할 수는 없다. 대신 그 핵심 사용층에서 MCP 도구 호출 570,373건이 남았다. 이 가운데 536,267건은 성공, 33,777건은 실패로 기록됐다. 성공 또는 실패 상태가 명시된 호출만 놓고 보면 성공률은 **94.1%**였다.

호출을 남긴 핵심 사용층은 연결 상태만 확인하고 떠나지 않았다. `create_text`는 105,643회, `set_fill_color`는 92,561회 성공했다. 둘만 합쳐도 198,204회다. 도구 이름을 생성·복제와 스타일·콘텐츠 편집으로 묶으면 390,642건, 전체 성공 호출의 **72.8%**였다.

**사용자가 원한 것은 단순한 연결이 아니라 AI가 Figma를 실제로 만지는 순간이었다.** MCP Magic의 역할은 그 순간까지 도달할 수 있는 입구를 만드는 일이었다.

### 넓게 발견됐고, 일부 작업 세션은 깊었다

GA4가 도달 범위와 호출 규모를 보여줬다면, Aptabase export는 관측된 작업 세션의 깊이를 보여줬다. 개발 빌드를 제외한 release 자료에서 MCP 호출이 들어간 세션은 전체 관측 세션의 일부였지만, 그 안에서는 긴 관측 span과 호출량의 긴 꼬리가 함께 나타났다. 아래 수치는 채택률이나 연속 사용 시간이 아니라, 남아 있는 세션 내부의 행동 깊이를 설명한다.

모든 사용자가 깊게 썼다고 말할 수는 없다. 대신 **발견은 넓었고, MCP가 포함된 관측 세션은 길었으며 일부 상위 세션에 호출이 강하게 집중됐다**고 말할 수 있다.

## 딸깍 다음에는 운영이 남았다

사용자가 늘자 질문이 달라졌다.

> 서로 다른 사용자와 운영체제, MCP 클라이언트, 네트워크 환경에서도 계속 믿고 쓸 수 있게 만들 수 있는가?

Yiseo가 운영하는 채널에는 정말 "딸깍"만 하면 디자인이 되는지, AI는 모두 무료인지, Cursor나 Claude Code 설정까지 앱이 대신해 주는지 묻는 질문이 들어왔다.

사용자가 말하는 "안 된다"의 원인은 모두 달랐다. 서버나 포트, 플러그인의 채널, MCP 클라이언트 설정, 운영체제 보안 중 어디에서든 연결이 막힐 수 있었다. GA4에는 포트 바인드 문제를 만난 활성 사용자 1,539명, 서버 시작 핸들러 중복 1,231명, 요청 시간 초과 1,056명이 기록됐다.

원인은 달라도 사용자에게는 같은 말로 도착했다.

> 안 된다.

호출 단위의 높은 성공률과 세션 단위의 마찰은 동시에 존재했다. GA4에서는 상태가 명시된 호출의 94.1%가 성공했지만, 별도 기간과 계측 방식의 Aptabase release 로그북에서는 MCP 세션 2,376개 중 1,424개(59.9%)에 실패 이벤트가 한 번 이상 남았다. 두 수치를 직접 비교할 수는 없다. 다만 관측된 MCP 세션 다수에 적어도 한 번의 실패 이벤트가 함께 남았다는 사실은 호출 단위 성공률만으로 작업 중 마찰을 설명할 수 없음을 보여준다.

첫 연결을 쉽게 만드는 것만으로는 충분하지 않았다. 그다음에는 사용자의 수준에 맞는 진단과 복구, 연결 안정성, 지원 운영이 필요했다. 이것은 부가적인 운영 업무가 아니라 첫 연결 다음에 만들어야 했던 제품 그 자체였다.

## 성공은 생존을 보장하지 않았다

우리는 언어 지원을 넓히고, 별도의 IDE 없이 로컬 LLM으로 작업하는 방식도 시험했다. 그러나 시간이 갈수록 큰 업데이트를 계속할 이유는 약해졌다. 사용량이 없어서가 아니라, 우리가 제품으로 만든 핵심 경험의 가장 자연스러운 자리가 공식 플랫폼 안으로 이동하기 시작했기 때문이다.

2026년 3월 Figma는 [`use_figma`를 통해 에이전트가 캔버스에 직접 쓰는 베타](https://www.figma.com/blog/the-figma-canvas-is-now-open-to-agents/)를 공개했고, 5월에는 [캔버스 안에서 바로 쓰는 Figma design agent](https://www.figma.com/blog/the-figma-agent-is-here/)를 발표했다. 문제 자체가 사라진 것이 아니었다. 그 문제를 푸는 기본 위치가 바뀌었다.

독립 도구는 플랫폼의 경계가 얇은 곳에서 새 수요를 먼저 발견하곤 한다. 수요가 충분히 분명해지면 파일, 권한, 디자인 시스템과 배포 면을 이미 가진 플랫폼은 그 흐름을 더 짧은 경로로 자기 표면 안에 넣을 수 있다. 사용자에게는 자연스럽고 반가운 변화지만, 서드파티에는 **기능의 성공과 독립 제품의 지속 가능성이 전혀 다른 문제**라는 뜻이다.

이것을 Figma가 우리를 보고 따라 했다는 이야기로 만들 수는 없다. 공식 기능이 사용자 감소를 일으켰다는 인과도 데이터로 증명하지 못한다. 다만 우리가 증명한 것은 실제 수요였고, 플랫폼은 그 수요를 더 자연스럽게 풀 수 있는 구조적 위치에 있었다.

그 방향이 맞다는 판단과 만든 사람의 감정은 별개였다. 문제를 충분히 일찍 발견해 제품으로 만들었고, 수만 명과 57만 건의 호출로 수요를 증명했다. 그러나 그다음에 우리가 오래 소유할 진단·복구, 교차 플랫폼 연결, 특정 팀의 깊은 워크플로를 더 빨리 정하지 못했다. 우리는 단순히 늦어서 밀려난 것이 아니라, 충분히 일찍 문제를 풀고도 다음의 지속 가능한 자리를 늦게 정했다.

그래서 MCP Magic은 종료보다 유지보수 모드에 가깝게 두려고 한다. 필요한 수정은 이어가겠지만, 지금의 형태로 큰 기능 업데이트를 계속할 이유는 이전보다 작아졌다.

## 다음에는 더 일찍 정할 것

돌이켜보면 첫 연결을 쉽게 만든 뒤의 결정을 오래 미뤘다.

첫째, **다음 목표를 더 빨리 좁혔어야 했다.** 설치 도우미로 남을지, 연결 진단과 복구 도구로 갈지, 반복 사용이 많았던 Figma 작업을 하나의 워크플로로 묶을지 정하지 못했다. 첫 번째 문제가 풀린 뒤에는 새로운 제품 가설이 필요했다.

둘째, **지원의 경계를 분명히 했어야 했다.** 어디까지가 앱의 책임이고 어디부터가 Figma 플러그인, MCP 클라이언트, 운영체제, AI 비용의 문제인지 더 명확해야 했다. 반복되는 "안 된다"를 자가 진단과 복구 흐름으로 바꾸는 일도 기능 개발만큼 중요했다.

셋째, **사용 데이터를 다음 가설에 더 일찍 연결했어야 했다.** 뒤늦게 데이터를 정리하고 나서야 넓은 발견, 호출이 집중된 일부 작업 세션, 높은 호출 성공률과 실패 이벤트가 함께 남은 세션이 동시에 보였다. 출시 초기부터 첫 연결, 첫 성공, 반복 사용과 실패 이후의 행동을 나눠 봤다면 업데이트의 우선순위도 달라졌을 것이다.

## 57만 번이 남긴 것

이 프로젝트가 증명한 것은 수요였지, 그 수요를 우리가 영구히 소유한다는 권리는 아니었다.

MCP Magic은 사용자를 0에서 80까지 데려가는 첫걸음을 "딸깍"으로 줄였다. 앱에는 31,360명의 활성 사용자가 도달했고 MCP 도구 호출은 570,373번 쌓였다. 그 호출 기록에는 텍스트를 만들고, 색을 입히고, 프레임을 만들고, 레이아웃을 다듬은 작업이 남았다.

동시에 마지막 20%에는 진단과 복구, 운영과 지원, 그리고 어디까지 책임질지에 대한 선택이 남았다. 공식 플랫폼이 같은 문제를 더 자연스럽게 풀기 시작한 뒤에는 무엇을 만들 수 있느냐보다 무엇을 오래 소유할 수 있느냐가 더 중요한 질문이 됐다.

이것은 실패의 기록도, 업데이트를 멈추는 변명도 아니다. **성공한 기능과 유지보수 모드에 들어간 제품은 동시에 존재할 수 있다.** 57만 건의 호출은 우리가 옳은 문제를 풀었다는 증거이면서, 다음에는 지속 가능한 자리를 더 일찍 정해야 한다는 청구서이기도 하다.
