> 우리가 만든 것은 새로운 AI 기능이 아니라, 설치에서 첫 성공 호출까지 포기하지 않고 건널 수 있는 입구였다.

GA4에는 **31,360명의 누적 사용자**와 **570,373건의 MCP 도구 호출**이 남았다. 숫자는 이 프로젝트가 실제로 쓰였다는 사실을 보여준다. 하지만 주형님과 진영님이 만든 Talk To Figma 데스크톱 앱을 MCP Magic 앱으로 함께 확장하고 출시하면서, 내게 더 오래 남은 것은 그다음의 질문이었다. 사용자가 늘어난 뒤에도 이 경험을 어떻게 안정적으로 이어 갈 것인가. 그리고 우리가 만든 길은 앞으로 어디에 남게 될 것인가.

Figma에서 AI가 디자인을 읽고 고치는 장면은 처음부터 매력적이었다. 다만 다른 사람에게 도구를 건네보면, 많은 이가 그 장면에 닿기도 전에 멈췄다. 런타임과 서버, MCP 클라이언트와 Figma 플러그인을 각각 설정해야 했고, 문제가 생길 때마다 터미널과 플러그인 사이를 오가야 했기 때문이다.

주형님과 진영님이 이 준비 과정을 하나의 데스크톱 앱에 담은 프로토타입을 보여줬을 때, 나는 "이거다" 싶었다. 내가 바랐던 건 거창한 자동화가 아니었다. 한국에서 유행하는 표현을 빌리면, **"딸깍"하면 서버가 뜨는 경험**이었다.

## 우리가 만든 것은 기능이 아니라 입구였다

MCP Magic은 빈 곳에서 시작한 프로젝트가 아니다. 출발점에는 Grab Design 팀에서 사용하다 오픈소스로 공개한 [Cursor Talk To Figma MCP](https://github.com/grab/cursor-talk-to-figma-mcp)가 있었다. 자연어로 Figma를 읽고 수정하는 핵심 기능은 이미 작동하고 있었다.

주형님이 먼저 본 문제는 기능 부족이 아니라, 사용자가 처음 성공하기까지의 복잡한 과정이었다. 2025년 7월, 주형님과 진영님은 서비스 실행·설정·연결 상태·로그를 트레이 메뉴에 모은 Talk To Figma 데스크톱 앱을 GitHub에 공개했다. 이 간단한 앱은 현재 Grab의 공식 오픈소스이자 MCP Magic의 백본이 됐다.

그 프로토타입을 계기로 나는 프로젝트에 적극적으로 참여했다. 데스크톱 UI의 아이디어를 함께 다듬고, 복잡한 설정을 사용자 언어로 풀어내는 방식과 채널을 통한 홍보를 맡았다. 같은 해 10월, 우리는 이 기반을 MCP Magic이라는 패밀리 앱으로 확장해 출시했다. 이후 사용자 의견을 들으며 여러 번 구조를 다시 썼지만, 붙들고 있던 질문은 바뀌지 않았다.

> 어떻게 해야 더 많은 사람이 첫 연결을 지나 실제 작업까지 도달할 수 있을까?

앱이 MCP 자체를 더 강력하게 만든 것은 아니다. 우리는 이미 있던 기능에 도달하는 길을 짧고 분명하게 만들었다. 시작하기 전에 포기한다면, 아무리 강력한 기능도 사용자에게는 없는 것과 같았으니까.

## 이 프로젝트는 실제 작업으로 이어졌다

GA4의 실제 기록이 있는 11개월 동안 `mcp_tool_call` 이벤트를 한 번 이상 남긴 활성 사용자는 1,633명이었다. 이 값은 앱 전체 누적 사용자 31,360명과 이벤트별로 따로 집계됐으므로, 모든 사용자가 MCP 작업까지 갔다고 말할 수는 없다. 다만 이 핵심 사용층에서 570,373건의 MCP 도구 호출이 남았다. 이 가운데 536,267건은 성공, 33,777건은 실패로 기록됐다. 성공 또는 실패 상태가 명시된 호출만 보면 성공률은 **94.1%**였다.

**이 1,633명은 AI와 Figma를 유료로 활용하는 프로 사용자들이다.** Cursor나 Claude Code 같은 AI 도구와 Figma 유료 플랜을 이미 업무에 쓰고 있었고, 그 위에서 MCP Magic까지 도입했다. 따라서 1,633명은 단순 설치나 무료 체험이 아니라, 새 워크플로를 실제 업무에 먼저 끌어들인 상당히 이른 초기 수용자층을 뜻한다.

이 기록에서 가장 반가웠던 것은, 사용자가 연결 상태만 확인하고 떠나지 않았다는 점이다. `create_text`는 105,643회, `set_fill_color`는 92,561회 성공했다. 둘을 합치면 198,204회다. 도구 이름을 생성·복제와 스타일·콘텐츠 편집으로 묶으면 390,642건으로, 전체 성공 호출의 **72.8%**를 차지했다.

**사용자가 원한 것은 단순한 연결이 아니라 AI가 Figma를 실제로 만지는 순간이었다.** 주형님과 진영님이 데스크톱 앱으로 연 가능성을, 우리는 MCP Magic에서 더 많은 사람이 실제 작업으로 이어 갈 수 있게 다듬었다.

### 깊이는 고르지 않았고, 일부 세션은 매우 깊었다

GA4가 도달 범위와 호출 규모를 보여줬다면, Aptabase export는 세션 안에서 MCP가 얼마나 깊게 쓰였는지를 보여줬다. 여기서 중요한 것은 전체 세션 수가 아니라, MCP를 쓰기 시작한 일부 세션의 밀도다.

관측된 MCP 작업의 깊이는 균등하지 않았다. 전형적인 세션은 관측 span 중앙값 25분 31초와 추정 호출 중앙값 11회였지만, 깊은 5%의 문턱에서는 2시간 12분 54초와 367회까지 올라갔다. 가장 긴 관측 세션은 12시간 36분 41초, 최대 추정 호출은 9,631회였고, 상위 5% 세션만으로 전체 추정 호출의 55.1%가 남았다. **MCP는 일부 사용자에게 연결을 시험하는 기능이 아니라, 길고 반복적인 디자인 작업의 도구가 됐다.**

## 딸깍 다음에는 운영이 남았다

사용자가 늘자, 내가 받는 질문도 달라졌다. 나는 Friends of Figma Seoul을 포함한 몇 개의 카카오톡 오픈채팅방을 운영한다. 합치면 수천 명이 활발하게 활동하는 채널들인데, 그곳에서 먼저 변화가 보이기 시작했다.

처음에는 신기해하는 반응이 많았다. 그러다 어느 순간 질문의 결이 달라졌다.

> "정말 딸깍만 하면 디자인이 되나요?"<br>
> "이건 무료인가요?"<br>
> "Cursor에는 어떻게 설치하나요?"

정확한 이유를 단정할 수는 없지만, AI나 MCP가 신기하지만 나와는 거리가 있는 일처럼 느껴지던 때는 빠르게 지나간 것 같았다. 불과 1년 사이에 많은 디자이너에게 이것은 업무에 바로 써야 할 도구가 됐고, 질문 사이에서는 급하게 배우고 써야 한다는 FOMO도 느껴졌다.

이 질문들은 결국 하나로 모였다.

> 서로 다른 사용자와 운영체제, MCP 클라이언트, 네트워크 환경에서도 계속 믿고 쓸 수 있게 만들 수 있는가?

사용자가 말하는 "안 된다"의 원인은 모두 달랐다. 서버나 포트, 플러그인의 채널, MCP 클라이언트 설정, 운영체제 보안 중 어디에서든 연결이 막힐 수 있었다. GA4에는 포트 바인드 문제를 만난 활성 사용자 1,539명, 서버 시작 핸들러 중복 1,231명, 요청 시간 초과 1,056명이 기록됐다.

원인은 달라도 사용자에게는 같은 말로 도착했다.

> 안 된다.

호출 단위의 높은 성공률과 세션 단위의 마찰은 동시에 존재했다. GA4에서는 상태가 명시된 호출의 94.1%가 성공했지만, 별도 기간과 계측 방식의 Aptabase release 로그북에서는 MCP 세션 2,376개 중 1,424개(59.9%)에 실패 이벤트가 한 번 이상 남았다. 두 수치를 직접 비교할 수는 없다. 다만 관측된 MCP 세션 다수에 적어도 한 번의 실패 이벤트가 함께 남았다는 사실은, 호출 단위 성공률만으로 작업 중 마찰을 설명할 수 없다는 점을 보여준다.

그때 깨달았다. 시작을 쉽게 만드는 것만으로는 충분하지 않았다. 사용자 수준에 맞는 진단과 복구, 연결 안정성, 지원 운영이 필요했다. 이는 부가적인 운영 업무가 아니라, 그다음에 만들어야 할 제품의 일부였다.

## 성공은 생존을 보장하지 않았다

MCP Magic을 운영하며 우리는 언어 지원을 넓히고, 별도의 IDE 없이 로컬 LLM(Google Gemma 4)으로 작업하는 방식도 시험했다. 그러나 시간이 갈수록 큰 업데이트를 이어 갈 이유는 약해졌다. 사용량이 없어서가 아니라, 우리가 제품으로 만든 핵심 경험이 공식 플랫폼 안으로 들어가기 시작했기 때문이다.

2026년 3월 Figma는 [`use_figma`를 통해 에이전트가 캔버스에 직접 쓰는 베타](https://www.figma.com/blog/the-figma-canvas-is-now-open-to-agents/)를 공개했고, 5월에는 [캔버스 안에서 바로 쓰는 Figma design agent](https://www.figma.com/blog/the-figma-agent-is-here/)를 발표했다. 문제 자체가 사라진 것이 아니었다. 그 문제를 푸는 기본 위치가 바뀌었다.

독립 도구는 플랫폼의 경계가 얇은 곳에서 새 수요를 먼저 발견하곤 한다. 수요가 충분히 분명해지면 파일, 권한, 디자인 시스템, 배포 면을 이미 가진 플랫폼은 그 흐름을 더 짧은 경로로 자기 표면 안에 넣을 수 있다. 사용자에게는 자연스럽고 반가운 변화지만, 우리 같은 서드파티에게는 **기능의 성공과 독립 제품의 지속 가능성이 전혀 다른 문제**라는 뜻이었다.

이것을 Figma가 우리를 보고 따라 했다는 이야기로 만들 수는 없다. 공식 기능이 사용자 감소를 일으켰다는 인과도 데이터로 증명하지 못한다. 다만 우리가 증명한 것은 실제 수요였고, 플랫폼은 그 수요를 더 자연스럽게 풀 수 있는 구조적 위치에 있었다.

그 방향이 맞다는 판단과, 함께 만든 사람으로서의 감정은 별개였다. 우리는 문제를 충분히 일찍 발견해 제품으로 만들었고, 수만 명과 57만 건의 호출로 수요를 증명했다. 하지만 그다음에 오래 소유할 진단·복구, 교차 플랫폼 연결, 특정 팀의 깊은 워크플로를 더 빨리 정하지 못했다. 단순히 늦어서 밀려난 것이 아니라, 다음의 지속 가능한 자리를 늦게 정한 셈이었다.

그래서 MCP Magic은 종료보다 유지보수 모드에 가깝게 두기로 했다. 필요한 수정은 이어가겠지만, 지금의 형태로 큰 기능 업데이트를 계속할 이유는 이전보다 작아졌다.

## 다음에는 더 일찍 정할 것

돌이켜보면, 시작을 쉽게 만든 뒤 다음에 무엇을 할지 결정하는 데 너무 오래 걸렸다. 이 프로젝트를 함께 만들며 남긴 메모는 세 가지다.

첫째, **다음 목표를 더 빨리 좁혔어야 했다.** 설치 도우미로 남을지, 연결 진단과 복구 도구로 갈지, 반복 사용이 많은 Figma 작업을 하나의 워크플로로 묶을지 정하지 못했다. 첫 번째 문제가 풀린 뒤에는 새로운 제품 가설이 필요했다.

둘째, **지원의 경계를 분명히 했어야 했다.** 어디까지가 앱의 책임이고 어디부터가 Figma 플러그인, MCP 클라이언트, 운영체제, AI 비용의 문제인지 더 명확히 했어야 했다. 반복되는 "안 된다"를 자가 진단과 복구 흐름으로 바꾸는 일도 기능 개발만큼 중요했다.

셋째, **사용 데이터를 다음 가설에 더 일찍 연결했어야 했다.** 뒤늦게 데이터를 정리하고 나서야 넓은 발견, 호출이 집중된 일부 작업 세션, 높은 호출 성공률과 실패 이벤트가 함께 남은 세션이 동시에 보였다. 출시 초기부터 시작, 첫 성공, 반복 사용, 실패 이후의 행동을 나눠 봤다면 업데이트 우선순위도 달라졌을 것이다.

## 57만 번이 남긴 것

이 프로젝트가 증명한 것은 수요였지, 그 수요를 우리가 영구히 소유한다는 권리는 아니었다.

MCP Magic은 사용자가 AI로 Figma를 실제로 다루는 지점까지 가는 첫걸음을 줄였다. 그 기록에는 텍스트를 만들고, 색을 입히고, 프레임을 만들고, 레이아웃을 다듬은 작업이 남았다.

동시에 마지막 20%에는 진단과 복구, 운영과 지원, 그리고 어디까지 책임질지에 대한 선택이 남았다. 공식 플랫폼이 같은 문제를 더 자연스럽게 풀기 시작한 뒤에는 무엇을 만들 수 있느냐보다 무엇을 오래 책임질 수 있느냐가 더 중요한 질문이 됐다.

이것은 실패의 기록도, 업데이트를 멈추는 변명도 아니다. **성공한 기능과 유지보수 모드에 들어간 제품은 동시에 존재할 수 있다.** 57만 건의 호출은 주형님과 진영님이 시작하고 내가 MCP Magic으로 함께 확장한 일이 옳은 문제를 풀었다는 증거다. 동시에 다음에는 지속 가능한 자리를 더 일찍 정해야 한다는 청구서이기도 하다.
