## 한 줄 결론

결과적으로 핵심 가치는 기능 추가보다 첫 실행과 첫 연결까지의 불안을 줄인 것이었다. GA4 데이터는 이 데스크톱 앱이 단순한 실험이 아니라, 많은 사용자가 실제로 실행하고 연결하고 MCP 도구를 호출한 제품이었다는 점을 보여준다.

## 잘됐던 점 후보

이 프로젝트는 “Figma에 기능을 하나 더 붙였다”보다 “Figma MCP를 실제로 써볼 수 있는 데스크톱 진입로를 만들었다”에 가깝다.

데이터상으로도 그 진입로는 작동했다. 31,360명의 활성 사용자가 있었고, MCP 도구 호출은 570,373건까지 쌓였다. 특히 호출 이벤트 중 성공으로 기록된 것이 536,267건, 실패로 기록된 것이 33,777건이라 성공/실패가 명시된 호출 기준 성공률은 약 94.1%다.

## 플랫폼과 사용 깊이

운영체제 기준 활성 사용자는 Windows가 21,090명, MacOS가 10,444명이다. 사용자 수로는 Windows가 더 컸다.

반면 MCP 호출 깊이는 MacOS 쪽이 더 강했다. 성공/실패가 기록된 MCP 호출 기준 MacOS는 366,518건, Windows는 203,526건이었다. 이 차이는 “Windows에서 더 넓게 퍼졌고, MacOS에서는 더 깊게 사용됐다” 정도로 조심스럽게 해석할 수 있다.

단, MacOS 이벤트에는 오류 이벤트도 크게 섞여 있으므로 “Mac 사용자가 더 열성적이었다”처럼 단정하면 위험하다.

## 운영 부담

이 프로젝트가 잘됐다는 사실과 별개로, 운영 부담도 데이터에 선명하게 남아 있다.

`app_exception`은 1,081,199건이고, 활성 사용자는 3,684명이다. 하지만 이 숫자를 그대로 “백만 건의 오류”로 말하면 과장될 수 있다. 예를 들어 `write EPIPE`는 897,378건이지만 활성 사용자는 12명뿐이고, `net::ERR_TUNNEL_CONNECTION_FAILED`는 99,996건이지만 활성 사용자는 1명뿐이다. 반복 오류가 이벤트 수를 크게 부풀렸다.

반대로 사용자 수 기준으로 의미 있는 운영 이슈도 있었다.

이 부분은 회고의 Problem 또는 Learning에 넣기 좋다. 사용자는 기능보다도 “연결 상태”, “포트 충돌”, “서버 재시작”, “타임아웃”, “로그 복사”, “이슈 리포트” 같은 운영 표면에서 불안을 느낀다.

## 왜 큰 업데이트 우선순위가 낮아졌나

데이터만 보면 프로젝트는 충분히 잘됐다. 31,360명의 활성 사용자, 570,373건의 MCP 도구 호출, 165개의 country 값, 37개의 언어 값은 작은 실험 이상이었다.

그런데 큰 업데이트를 계속 밀어붙일 이유는 약해졌다. 이 프로젝트가 해결하려고 했던 핵심 경험, 즉 Figma를 AI/MCP 워크플로우와 연결하는 경험이 Figma 공식 MCP나 Canvas용 Agent 같은 공식/플랫폼 계층으로 들어왔기 때문이다.

따라서 회고의 결론은 “실패해서 접는다”가 아니다. 더 정확히는 “프로젝트가 증명한 필요가 플랫폼으로 흡수되었고, 개인 프로젝트가 계속 큰 업데이트를 감당할 이유가 줄었다”에 가깝다. 감정적으로도 힘들어서, 큰 업데이트를 이어가기 어렵다는 문장도 간단하게 넣을 수 있다.

## KPT 반영 후보

Keep:

- 첫 실행부터 첫 연결까지의 불안을 줄이는 데 집중한 점.
- 사용자가 실제 디자인 편집 MCP 작업을 수행할 수 있을 만큼 연결 경험을 제품화한 점.
- Google Analytics MCP로 실제 사용 규모, 국가 분포, OS 분포, MCP 호출 성공률, 온보딩 이벤트를 확인할 수 있게 해둔 점.
- 데스크톱 앱을 단순 테스트 환경이 아니라 설치하고 실행하는 제품으로 만든 점.

Problem:

- 기능 자체가 플랫폼 공식 기능으로 흡수되면 개인 프로젝트가 계속 같은 강도로 업데이트하기 어렵다.
- 오류 이벤트가 매우 크게 쌓였고, 일부는 적은 사용자에게서 반복 발생해 데이터 해석을 어렵게 만들었다.
- 연결/포트/서버 상태/OAuth/로그 같은 운영 표면이 생각보다 큰 제품 영역이었다.
- GA4 이벤트 설계에 `(not set)`이 많아 일부 세그먼트 분석이 제한됐다.

Try/Learning:

- 다음에는 “기능”보다 “플랫폼이 흡수하기 어려운 사용 맥락”을 더 명확히 잡는다.
- 이벤트 설계에서 성공, 실패, 첫 연결 완료, 재연결, 설정 복사, 진단 액션을 더 엄격하게 분리한다.
- 오류 이벤트는 이벤트 수만 보지 말고 영향을 받은 활성 사용자 수를 함께 보도록 설계한다.
- 큰 업데이트를 계속할지 판단할 때 사용량뿐 아니라 플랫폼 흡수 가능성, 운영 부담, 개인적 지속 가능성을 같이 본다.

## 회고 본문 문장 후보

- 결과적으로 핵심 가치는 기능 추가보다 첫 실행과 첫 연결까지의 불안을 줄인 것이었다.
- 1년 동안 31,360명의 활성 사용자가 있었고, MCP 도구 호출은 570,373건까지 쌓였다.
- 사용자는 앱을 켜보기만 한 것이 아니라, 텍스트를 만들고 색을 바꾸고 프레임을 만들고 노드를 조회하는 실제 디자인 작업을 수행했다.
- 한국에서 만든 작은 데스크톱 앱이었지만, GA4 기준 165개의 country 값과 37개의 언어 값이 잡혔다.
- 잘된 프로젝트였기 때문에 더 아쉽다. 다만 핵심 경험이 공식 플랫폼 기능으로 흡수되면서, 개인 프로젝트로 큰 업데이트를 계속할 이유는 줄어들었다.
- 실패해서 내려놓는 것이 아니라, 증명한 필요가 더 큰 플랫폼으로 이동한 뒤 남는 허무함을 받아들이는 쪽에 가깝다.

## 주의해서 써야 할 데이터

- `eventCount` 전체 1,826,813건은 `app_exception` 1,081,199건을 포함하므로 순수 사용량으로 말하면 안 된다.
- 국가 값 165개에는 `(not set)`이 포함된다. 정확한 국가 수처럼 단정하지 않는다.
- 지역/도시 데이터는 `(not set)`이 많아 세부 지리 분석에는 쓰지 않는다.
- 브라우저, `customEvent:platform`, `customEvent:os_info`는 대부분 `(not set)`이라 강한 근거로 쓰지 않는다.
- GA4 funnel 수치는 순서 기반 분석이므로 전체 이벤트 합계와 직접 비교하지 않는다.
- `averageSessionDuration`은 데스크톱 앱/텔레메트리 맥락에서 비정상적으로 커 보이므로 핵심 자랑 지표로 쓰지 않는다.

## 추가 분석이 필요하지만 현재는 보류할 질문

## 반복 분석 로그

1. 전체 규모, 이벤트, 국가, 언어, OS, 앱 버전, MCP 호출, 성공/실패, 온보딩, 오류, 월별/주별 추이를 확인했다.
2. 액션/라벨 축을 추가로 확인해 첫 연결 제품 표면을 뒷받침하는 근거를 보강했다.
3. 유입 축은 `direct / none` 또는 `not set`으로만 잡혀 회고용 근거로 쓰지 않기로 했다.
4. 화면 제목/경로 축에서 첫 연결 튜토리얼 단계와 오류/로그 화면 사용을 확인했다.
5. 브라우저, 화면 해상도, 신규/재방문, referrer, channel group, stream, `customEvent:value`, `customEvent:duration_ms`를 추가 검산했다. 새 발표용 인사이트는 없었고, 대부분 “쓰지 말아야 할 축”으로 분류했다.
6. 다시 보조 유입/스트림 축을 확인했지만 새 인사이트가 나오지 않았다. 종료 조건의 “2회 연속 새 발표용 인사이트 없음”을 충족했다.
